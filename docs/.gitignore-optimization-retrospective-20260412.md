### 典型问题解决过程

**问题**：重复规则导致文件混乱

**根因分析**：项目迭代过程中，不同开发者多次添加相似规则，累积导致重复。例如：
- Python初始添加了 `*.so`，后来C扩展又添加了一次
- Django添加了 `*.log`，后来日志分类又添加了一次

**解决方案**：
1. 系统性扫描所有规则，去重
2. 按类别重组，将同类规则放在一起
3. 删除重复，保留一份

**经验**：定期整理 `.gitignore` 是良好习惯，随着项目发展规则会累积。

## 9. 经验方法

### 成功要素
1. **Spec驱动**：先分析问题，制定规范，批准后再执行修改，减少盲目性
2. **逐文件验证**：对每个需要判断的文件夹，都：
   - 看目录内容 → 读示例文件 → 读源代码 → 分析性质 → 对比类似文件夹 → 得出结论
3. **分类决策准则**：建立清晰的判断标准，不凭感觉决策
4. **保持一致性**：与项目已有实践保持一致（已经忽略 `wallets/`，同类的 `session-keys/` 也应该忽略）

### `.gitignore` 优化最佳实践

#### 五步检查法
```
1. 去重：扫描所有规则，移除重复
2. 补全：添加操作系统和IDE的标准规则
3. 分类：按逻辑类别分组排序
4. 判断：逐个文件夹分析，应用决策准则
5. 验证：确认重要文件没有被误忽略
```

#### 判断决策树

```
开始
  │
  ├─→ 是否包含私钥/凭证/敏感信息？
  │       ├─ 是 → 应该忽略 ⟲
  │       └─ 否 → 继续
  │
  ├─→ 是否是运行时动态生成，持续变化？
  │       ├─ 是 → 应该忽略 ⟲
  │       └─ 否 → 继续
  │
  ├─→ 是否每个开发者/部署都不同？
  │       ├─ 是 → 应该忽略 ⟲
  │       └─ 否 → 继续
  │
  └─→ 是源码/配置/业务需要共享吗？
          ├─ 是 → 不应该忽略 ✅
          └─ 否 → 应该忽略 ⟲
```
